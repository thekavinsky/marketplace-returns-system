# Use Cases

## UC-01. Создание заявки на возврат

**Основной участник:** покупатель.

**Предусловия:** покупатель авторизован, заказ существует и был получен.

### Основной сценарий

1. Покупатель выбирает заказ и позиции для возврата.
2. Для каждой позиции он указывает количество и причину.
3. Покупатель выбирает способ передачи товара.
4. Система запрашивает данные заказа в Order Service.
5. Система проверяет владельца заказа, дату получения и количество.
6. Если проверки пройдены, система создаёт заявку.
7. Заявка получает статус `APPROVED`.
8. Система создаёт обратную доставку.
9. Покупателю показывается номер заявки и дальнейшие действия.

### Альтернативные сценарии

- Если заказ принадлежит другому пользователю, система возвращает ошибку доступа.
- Если срок возврата закончился, заявка получает статус `REJECTED` и причину отказа.
- Если указано слишком большое количество, система не создаёт заявку и показывает ошибку поля.
- Если Order Service недоступен, система сообщает о временной ошибке.

**Результат:** заявка создана и сохранена либо пользователь получил понятную причину ошибки.

## UC-02. Отмена заявки

1. Покупатель открывает активную заявку.
2. Нажимает «Отменить возврат».
3. Система проверяет, что товар ещё не передан.
4. Заявка получает статус `CANCELLED`.
5. Система отправляет отмену в Delivery Service.

Если товар уже находится в пути, отмена недоступна.

## UC-03. Проверка товара на складе

1. Склад получает товар и открывает заявку.
2. Сотрудник указывает полученное и принятое количество.
3. Если товар не принят, сотрудник указывает причину.
4. Система считает сумму возврата.
5. При положительной сумме создаётся запрос в Payment Service.
6. После успешной оплаты заявка получает статус `COMPLETED`.

Если склад не принял ни одной позиции, заявка завершается со статусом `COMPLETED_WITHOUT_REFUND`.
